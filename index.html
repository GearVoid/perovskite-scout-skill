现在 Agent 到底怎么赚钱？

从主流玩家的定价方式，看 Agent 商业化的几种路径。最近很多平台都在讲 Agent，但真正值得看的不是概念，而是它们已经开始怎么收费、向谁收费、靠什么持续收费。

是卖账号，还是卖使用量？
是卖 Agent，还是卖服务包？
单独卖 Skill，为什么很难形成壁垒？

四个已经出现的收费信号

Seat 账号订阅跟着企业软件采购走。
Usage 对话、任务、动作用得越多，账单越高。
Credit token、credit、工具调用底层能力按消耗收费。
Service 部署、实施、行业方案把 Agent 接进真实业务。
01 / 一句话结论

现在 Agent 赚钱，主要不是靠卖 Skill。

Agent 本身越来越容易做，难收费的是“一个 Agent”；容易收费的是“Agent 解决了一个持续发生的问题”。

目前已经跑起来的收费方式，主要集中在几类。

不是所有模式都已经成熟，但价格信号已经比较清楚：大厂把 Agent 放进账号、用量、底层资源和服务交付里收费。单独卖一个 Skill，反而不是主流路径。

  • 01按账号订阅
  • 02按对话、任务、动作收费
  • 03按 credit、token、工具调用收费
  • 04按企业部署和实施服务收费
  • 05按行业方案收费

04 / 哪种模式最可能赚钱

越靠近业务结果，越容易卖高价；越像通用工具，越容易卷价格。

单卖 Skill 很难。卖“帮客户把一件事做成”的方案，才更接近真实收入。

免费工具 / Prompt 门槛低，容易复制。
Skill 模板 可以卖小钱，但很难长期防守。
通用 Agent 用户多，但付费意愿有限。
企业流程 Agent 开始有预算，因为接近流程效率。
行业解决方案 更接近客户痛点，也更需要行业经验。
部署 + 持续服务 短期最容易收高价，但重交付。
05 / 为什么单卖 Skill 或 Agent 门槛低

Skill 可以复制，但客户场景复制不了那么快。

一个 Skill，本质上是一套工作流说明。只要别人看懂逻辑，就可能做出类似版本。

所以问题不在“能不能写一个 Skill”，而在它有没有接到客户的数据、接口、流程、权限和交付服务。接不上这些，它更像模板；接上以后，才可能变成方案。

单卖 Skill 的问题

客户买到的只是“怎么做”的说明，还没有买到结果。只要平台或竞品看懂流程，就能做出相近版本。

  • 只能按模板或小工具定价，很难卖到企业预算。
  • 流程逻辑一旦暴露，复制成本通常低于获客成本。
  • 用户会问：我为什么不自己写一个，或者等平台内置？
  • 平台一旦把同类能力做进产品，独立 Skill 很容易被替代。
  • 缺少数据、权限和复核链路，就很难承诺业务结果。

从模板变成方案，需要绑定这些东西

Template to Solution

真正的门槛不在 prompt，而在把它接进客户现场：数据能不能拿到，系统能不能连上，失败以后谁来复盘。

  • 客户数据：资料、历史工单、合同、知识库、业务记录。
  • 系统接口：CRM、ERP、客服、审批、文档和内部工具。
  • 业务流程：输入从哪里来，输出给谁，异常怎么处理。
  • 权限规则：哪些动作能自动做，哪些必须人工确认。
  • 交付服务：部署、培训、监控、迭代和失败样本复盘。
  • 行业经验：知道哪些判断不能只交给模型。
06 / 对我们有什么启发

如果未来要做，先不要想着卖 Skill。

真正要判断的是：有没有一个足够具体、足够高频、客户愿意付钱的问题。

更现实的路径：先找一个具体业务问题，做成可用工具，证明能省时间或减少错误，再包装成方案。做一堆 Skill 放出去等别人付费，这条路很难。

Direction A

内部效率工具

目的：先验证能力，不急着赚钱。它适合积累经验，但不应该被误认为商业化已经成立。

Direction B

企业定制方案

目的：帮客户接数据、改流程、做部署，靠项目收费。它更容易拿到钱，但交付会更重。

Direction C

行业场景产品

目的：选择一个高频场景，做成可复制产品，靠订阅加服务收费。它最需要行业判断。
